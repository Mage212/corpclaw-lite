# Roadmap

Планы развития CorpClaw Lite после релиза v0.1.0.

## v0.1.1 — Планировщик и редакторы

### SchedulerService (ядро)
Единый фоновый asyncio-сервис для cron-задач и календарных напоминаний. Polls SQLite каждые ~30с. Включает:
- **CronTasksWorker** — создаёт AgentLoop для выполнения задач по расписанию
- **CalendarWorker** — отправляет Telegram-напоминания (детерминистически, без LLM)
- Хранение в `data/scheduler.db`
- Стартует в Telegram orchestrator, shutdown через GracefulShutdown

### Cron-задачи
Пользователь настраивает задачи через чат: «каждый понедельник в 9:00 отправляй отчёт». Агент сохраняет расписание, SchedulerService выполняет.

### Календарь с напоминаниями
Встроенный календарь: пользователь создаёт события, агент присылает напоминания в Telegram.

### Инструменты редактирования данных
- **csv_editor** — редактирование CSV файлов (добавление/удаление строк и столбцов)
- **xlsx_editor** — редактирование Excel файлов (без пересоздания всего файла)
- **http_request** — расширенный HTTP-клиент (POST, PUT, DELETE, headers, auth) для интеграции с внешними API

## v0.x — Новые скиллы

- **project_assistant** — управление задачами, чеклисты, таймлайны. Интеграция с Jira/Confluence/GitLab через http_request. Для product, it, admin

- **archive_tool** — упаковка/распаковка ZIP и RAR архивов

## Потенциальные улучшения

Идеи для развития проекта, которые могут быть реализованы в будущем. Не привязаны к конкретной версии.

### Мультиканальность

Сейчас CorpClaw Lite поддерживает два канала: Telegram (основной) и CLI (отладочный). Оба работают по модели «агент получает сообщение и отвечает». Для корпоративного сегмента нужна возможность подключать произвольные мессенджеры и системы обмена сообщениями.

**Архитектурное решение:** вместо входящих HTTP-вебхуков (требуют публичного IP/порта) используется WebSocket/long-polling — агент сам инициирует исходящее соединение (pull-модель). Это критично для закрытых корпоративных сетей и VPN, где невозможно или запрещено открывать входящие порты.

**Реализация:** новый базовый класс `WebSocketChannel(Channel)` для каналов с push/pull моделью. Подключение к существующей системе каналов (`channels/base.py`). Каждый новый канал — отдельная реализация с минимальным boilerplate.

### Изоляция секретов (Vault)

Сейчас API-ключи и токены прокидываются через `.env` файл и доступны всем компонентам. При компрометации Docker-контейнера субагента злоумышленник получает доступ ко всем секретам хоста.

**Решение — Vault-подход:**
1. Секреты хранятся только в защищённом хранилище на хосте (зашифрованный файл или keychain)
2. Субагент запрашивает токен через HMAC-защищённый IPC-канал уже существующий в проекте
3. Токен существует только в RAM процесса и только на время выполнения конкретного инструмента
4. После выполнения — токен удаляется из памяти

**Что не меняется:** IPC Auth (HMAC+nonce) уже реализован — расширяется его применение для передачи секретов.

### Email-канал

Подключение агента к выделенному корпоративному почтовому ящику (например, `assistant@company.local`).

**Как работает:**
- IMAP polling — агент периодически проверяет входящие письма
- SMTP — отправка ответов и уведомлений
- Обработка вложений — PDF, Excel, Word автоматически анализируются
- Команды в теме/теле — пользователь пишет письмо с задачей, агент выполняет и отвечает

**Кейсы:** автоматизация обработки входящих заявок, отчётов от подрядчиков, сортировка и маршрутизация писем по департаментам.

### Корпоративные стандарты

Агент генерирует код и документы на основе «усреднённых» знаний LLM. В корпоративной среде нужны специфичные правила: стиль кода компании, требования к документации, регуляторные ограничения.

**Как работает:**
- Файлы стандартов хранятся в `config/standards/` (Markdown)
- Перед выполнением задачи скилл/субагент подгружает релевантные стандарты в контекст
- Результат генерации соответствует корпоративным правилам, а не общим рекомендациям LLM

**Примеры:** Java code style компании, требования к API-документации, шаблоны договоров, правила именования в Git.

### Интерактивный onboarding-wizard

Сейчас начальная настройка требует ручного редактирования `settings.yaml`, `departments.yaml` и `.env`. Для новых администраторов это порог входа.

**Что добавляется:**
- Команда `uv run corpclaw-lite onboard --wizard`
- Диалоговый CLI: подключение к Ollama → проверка модели → настройка департаментов → создание первого пользователя
- Автогенерация `settings.yaml` и `.env` на основе ответов
- Проверка связи с LLM на каждом шаге
