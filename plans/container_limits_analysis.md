# Анализ архитектуры: Контейнеризация vs Выполнение на хосте (v2 vs v1)

## Summary
Архитектурный анализ эффективности Docker-изоляции инструментов в CorpClaw Lite (Sandbox Protocol) по сравнению с прямым выполнением `subprocess.Popen` на стороне хоста из оригинального CorpClaw (v1). Документ описывает, почему выбранный подход безопаснее, и предлагает механизмы для контроля серверной нагрузки и распараллеливания задач в будущем.

## 1. Эффективность изоляции (Lite v2 vs v1)

В версии **v1** выполнение скриптов агента происходило прямо на хосте:
- **Риски безопасности (RCE):** LLM (или злоумышленник через prompt injection) могла выполнить деструктивные команды (например, `rm -rf /`) или прочитать токены из `config.yaml`.
- **Нестабильность хоста:** Скрипты с утечками памяти (Memory Leak) или бесконечными циклами (100% CPU) могли «повесить» весь сервер, блокируя работу других пользователей в Telegram.

В **Lite v2** мы используем паттерн **Sandbox Protocol** (`docker exec ... python -m agent_worker`):
- **Безопасность (10/10):** Контейнеры запускаются без сети (deny-by-default), с Read-Only файловой системой (кроме `/workspace` и `/tmp`) и под не-root пользователем. Выход за пределы песочницы заблокирован.
- **Изоляция:** Сбой или зависание инструмента изолированы внутри песочницы пользователя. Ошибки агента возвращаются в виде локализованных `SIGKILL` (Out Of Memory от ядра Linux).
- **Компромисс (Trade-off):** Незначительный оверхед на `docker exec` (~150–200 миллисекунд). Учитывая оптимизации `Lazy Loading`, этот оверхед с лихвой окупается стабильностью.

## 2. Контроль количества одновременных процессов и очередей

Главное опасение: *потеря контроля над количеством активных Python-процессов.*
Фактически, Docker в связке с асинхронным Python (`asyncio`) делает этот контроль **точнее и проще**.

**Почему мы не теряем контроль:**
1. **Единая точка входа:** Любая команда внутри песочницы проходит через `ContainerIPC.send_tool_call()`. Агентское приложение (хост) на 100% контролирует, сколько `docker exec` процессов оно прямо сейчас ожидает.
2. **Semaphore-очередь (План на будущее):** В `ContainerManager` или на уровне `AgentLoop` можно добавить управление очередями через `asyncio.Semaphore(MAX_CONCURRENT_TASKS)` (например, значение `MAX=10`).
   - Если 11-й пользователь отправляет задачу, хост просто заморозит корутину в очереди ожидания `await semaphore.acquire()`, **не порождая** новый процесс Docker-а до освобождения слота.
3. **Защита от Fork-бомб:** В `v1` скрипт мог плодить процессы, пока хост не падал. В `v2` мы опираемся на Cgroups (через `--pids-limit`), ограничивая создание дочерних процессов ядром Linux на уровне конкретного контейнера.

## 3. Управление железом (Cgroups) / Будущие задачи

В будущем архитектура позволяет использовать лимиты ресурсов Docker для надежного биллинга и приоритизации:

- [ ] В `policies.py` жестко зафиксировать лимиты Cgroups:
  ```python
  host_config = io_docker.create_host_config(
      mem_limit="512m",   # Жесткий лимит памяти на пользователя
      cpu_quota=50000,    # Жесткий лимит CPU (50% одного ядра)
      pids_limit=64       # Защита от fork-бомб скриптами
  )
  ```
- [ ] Внедрить глобальный `asyncio.Semaphore(MAX_CONCURRENT_TOOLS)` для ограничения числа параллельных вызовов инструментов `exec_script`.
- [ ] Внедрить `Queue` маршрутизацию для долгих задач, чтобы гарантированно распределять ресурсы между отделами (`departments.yaml`).

## Вывод
Архитектура с `corpclaw_agent_worker` + `ContainerIPC` перевела контроль ресурсов с уровня непредсказуемой ОС в предсказуемую песочницу. Это позволило нам управлять очередями на уровне `asyncio` внутри хост-приложения, получив бесплатный мониторинг и жесткие лимиты (cgroups) от движка Docker.
